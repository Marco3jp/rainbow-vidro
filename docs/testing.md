# テスト戦略

## 0. 基本方針

- **テスト必須**: 機能追加・変更には必ず対応するテストを追加する (テスト不要な場合は理由を PR 本文に書く)
- **決定論性に依拠**: ゲームロジックは固定シード + 固定 tick で完全再現可能なため、ロジックの単体・統合テストは決定論的に書ける
- **層ごとのテスト戦略**: ロジック層は単体テスト中心、UI 層は E2E、レンダラ層は最小限のスモーク

## 1. テストフレームワーク

| 用途 | フレームワーク |
| --- | --- |
| 単体・統合テスト | Vitest |
| E2E テスト | Playwright |
| 手動・探索的テスト | オートプレイ + 開発サーバ |

## 2. レイヤ別の戦略

### Core 層 (`src/core/`)

- **単体テスト**: 各 system 関数 / エンティティロジックを純粋関数的にテスト
- **統合テスト**: World 全体を回し、シナリオベースで検証 (例: 「シードSで100tick回したらブロックがN個破壊される」)
- 副作用は注入で差し替え (`SeededRng`, `Clock`)

### Render 層 (`src/render/`)

- 単体テストは最小限 (DisplayObject の生成/破棄など)
- 視覚的検証は E2E と手動確認

### Application / UI 層

- 主要シナリオ (タイトル → 開始 → 結果) を Playwright で1本通す
- HUD のキー要素 (HP/MP) が表示されることを E2E で確認

## 3. オートプレイによる Fuzz テスト

- 単純戦略でゲームを最後まで動かし、エラーが起きないかを検証
- 大量シードで実行 (例: 100 シード × プレイ完了) し、すべて完走することを確認
- ヘッドレス (Node.js) で実行可能にし、CI に組み込む (将来)

## 4. リプレイによるリグレッションテスト

- バグ報告時のシード + 入力ログを保存
- 修正後にそのリプレイを再生して、期待される結果を満たすことを確認
- リプレイベースのスナップショットテスト (フレームXの状態を JSON でスナップショット) も検討

## 5. CI で実行するチェック

- Lint (Biome)
- Type check (`tsc --noEmit`)
- Unit / Integration テスト (Vitest)
- Build (Vite)
- (将来) E2E (Playwright headless)
- (将来) Fuzz (オートプレイ × N シード)

すべて緑でなければマージ不可。

## 6. テストデータ規約

- 「マジックシード」をテスト内に埋め込まない場合は、定数として `tests/fixtures/seeds.ts` に集約する
- スナップショットテストは `tests/__snapshots__/` 配下
- リプレイログのフィクスチャは `tests/fixtures/replays/` 配下

## 7. ローカル実行コマンド (予定)

> Phase 1 の F-01 ~ F-04 完了後にこのセクションを実コマンドで更新する。

```bash
npm install        # 依存インストール
npm run dev        # 開発サーバ
npm run build      # 本番ビルド
npm run lint       # Biome lint
npm run test       # Vitest
npm run test:e2e   # Playwright E2E
```
